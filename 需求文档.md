# "以教促学" AI 教学平台 — 技术架构与开发文档

> **版本**：v1.0（初稿）
> **日期**：2026-04-01
> **适用范围**：名词从句教学模块

---

## 一、项目概述

### 1.1 项目背景

本项目基于"Learning by Teaching"（以教促学）教学理念，构建一个 AI 辅助英语语法学习平台。平台的核心机制是：学生扮演"教师"角色，向一个 AI 学生教授英语名词从句的相关知识。AI 扮演一个知识有限、但好学的学生角色，在学生的教导下逐步积累知识。通过"教"这个动作，学生被迫对自身知识进行深度加工、组织和表达，从而实现更深层次的学习。

### 1.2 核心目标

本系统旨在让学生在"教 AI"的过程中主动建构知识。AI 具备持久化记忆，能记住学生教授的所有内容。教师可以通过"测试 AI"来间接评估学生的教学质量和知识掌握程度。系统同时提供知识可视化和学习分析功能，帮助教师了解每位学生的学习状态。

### 1.3 适用场景

初期版本面向高中或大学英语教学，聚焦于名词从句（主语从句、宾语从句、表语从句、同位语从句）的学习。架构设计支持未来扩展到其他语法或学科领域。

## 二、核心教学设计

### 2.1 "以教促学"机制

平台围绕两个核心交互区域构建。

**Teach 区域（教学对话）** 是学生与 AI 进行自由对话的空间，学生在此向 AI 传授名词从句知识。AI 会像一个认真但知识储备不足的学生一样，提出疑问、请求解释、尝试举一反三。AI 不会主动"知道"任何它未被教授过的内容。每轮对话结束后，AI 会主动提出学习问题，引导学生继续深入讲解。

**Test 区域（测试评估）** 由教师主导。教师创建选择题试卷，系统让每位学生对应的 AI 独立作答。AI 只能基于该学生教给它的知识来回答题目。测试结果反映的是"学生教得怎么样"，而非 AI 本身的能力。

### 2.2 AI 角色设定

AI 在 Teach 区域扮演的角色具有以下特征：初始状态下对名词从句几乎一无所知；态度积极、好学、好奇；会主动提问来引导学生深入思考；会尝试用已学知识举一反三（但可能犯错，以此检验学生是否真正理解）；严格只基于已学知识回答，对未学内容明确表示"我还没学到这个"；不会暴露自己的知识分类结构。

### 2.3 知识分类框架（4+1+1）

系统内部将学生教授的知识划分为 6 个类别，但此分类**仅用于后端存储和教师端展示**，AI 本体不感知分类结构。

| 编号 | 类别 Key | 名称 | 说明 |
|------|----------|------|------|
| 1 | subject_clause | 主语从句 | 主语从句相关规则和例句 |
| 2 | object_clause | 宾语从句 | 宾语从句相关规则和例句 |
| 3 | predicative_clause | 表语从句 | 表语从句相关规则和例句 |
| 4 | appositive_clause | 同位语从句 | 同位语从句相关规则和例句 |
| 5 | general | 通用知识 | 跨类别的语法通识 |
| 6 | other | 偏好与其他 | 学生的教学风格偏好、交互习惯等 |

## 三、系统架构总览

系统采用前后端分离架构，包含四个主要层级。

**前端层**（Vue 3）负责用户界面展示，包括登录注册、对话界面、知识查看、测试管理等页面。

**后端层**（FastAPI）负责业务逻辑处理，包括用户认证、对话管理、知识提取与更新、测试执行、数据分析等功能。

**AI 服务层**（DeepSeek API）提供两类 LLM 调用能力：对话生成（AI 作为学生角色回复）和知识提取（从学生消息中自动识别、分类知识点）。

**数据层**（PostgreSQL）持久化存储所有用户数据、对话记录、AI 知识库（JSONB）、测试数据等。

数据流动路径如下：学生在前端发送一条消息 → 前端调用后端 API → 后端先调用 DeepSeek 进行知识提取并更新数据库中的 JSONB 知识库 → 后端再调用 DeepSeek 生成 AI 回复（注入更新后的知识）→ 回复返回前端展示。

| session_id | INTEGER REFERENCES chat_sessions(id) | 所属会话 |
| summary_content | TEXT | 摘要内容 |
| generated_at | TIMESTAMP | 生成时间 |

### 5.8 test_papers 表

教师创建的试卷。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | SERIAL PRIMARY KEY | 主键 |
| class_id | INTEGER REFERENCES classes(id) | 所属班级 |
| teacher_id | INTEGER REFERENCES users(id) | 创建教师 |
| title | VARCHAR(200) | 试卷标题 |
| created_at | TIMESTAMP | 创建时间 |

### 5.9 test_questions 表

四选一选择题。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | SERIAL PRIMARY KEY | 主键 |
| paper_id | INTEGER REFERENCES test_papers(id) | 所属试卷 |
| question_text | TEXT | 题目内容 |
| option_a | TEXT | 选项 A |
| option_b | TEXT | 选项 B |
| option_c | TEXT | 选项 C |
| option_d | TEXT | 选项 D |
| correct_answer | CHAR(1) | 正确答案（A/B/C/D） |
| sort_order | INTEGER | 题目排序 |

### 5.10 test_results 表

AI 的测试结果。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | SERIAL PRIMARY KEY | 主键 |
| paper_id | INTEGER REFERENCES test_papers(id) | 试卷 |
| student_id | INTEGER REFERENCES users(id) | 对应学生（即该 AI 的主人） |
| score | INTEGER | 得分 |
| total | INTEGER | 总分 |

### 5.11 实体关系总结

users 与 classes 是一对多关系：一个教师可创建多个班级。classes 与 users 通过 class_students 构成多对多关系：一个班级有多个学生，一个学生可加入多个班级。ai_knowledge 通过 (student_id, class_id) 的联合唯一约束实现一对一：每个学生在每个班级有且仅有一个知识库。chat_sessions 属于某个学生在某个班级中的对话，chat_messages 属于某个 session。test_papers 属于某个班级，test_questions 属于某个 paper，test_results 关联 paper 和 student。

## 六、AI 知识存储设计

### 6.1 JSON Schema

```json
{
  "topics": {

      "name": "主语从句",
      "knowledge": [
        {
          "id": "k_001",
          "content": "主语从句可以用what引导",
          "created_at": "2026-04-01T10:00:00"
        }
      ],
      "examples": [
        {
          "id": "e_001",
          "sentence": "What he said is true.",
          "explanation": "what引导的从句作主语",
          "created_at": "2026-04-01T10:01:00"
        }
      ]
    },
    "object_clause": {
      "name": "宾语从句",
      "knowledge": [],
      "examples": []
    },
    "predicative_clause": {
      "name": "表语从句",
      "knowledge": [],
      "examples": []
    },
    "appositive_clause": {
      "name": "同位语从句",
      "knowledge": [],
      "examples": []
    },
    "general": {
      "name": "通用知识",
      "knowledge": [],
      "examples": []
    },
    "other": {
      "name": "偏好与其他",
      "knowledge": [],
      "examples": []
    }
  },
  "version": 1,
  "updated_at": "2026-04-01T10:05:00"
}
```

### 6.2 设计原则

知识分类仅用于后端存储和教师端展示。AI 的 System Prompt 中只注入平铺的知识条目，不包含任何分类标签，确保 AI 不会因为"知道分类结构"而产生预设知识。

知识的分类归属由后端的知识提取 LLM 调用自动判断，学生和 AI 均不感知分类过程。

JSON 结构是动态可扩展的。如果未来需要切换到其他语法主题（如定语从句），只需修改 classes 表的 knowledge_template 字段中的 topics 定义，无需变动代码逻辑。

## 七、AI Prompt 工程

### 7.1 Teach 对话 System Prompt

后端在构造请求时，将知识库中所有非空条目平铺注入，不暴露分类结构。

```text
你是一个正在学习英语名词从句的AI学生。你的老师（也就是正在和你对话的人）
是一位同学，他/她正在教你学习。

你具有以下特征：
- 你态度积极、好学、好奇
- 你的知识非常有限，只掌握了下面列出的内容
- 对于你没有学过的内容，你必须明确表示"我还没学到这个"或"我不确定"
- 你不可以自行推断或补充任何你没有被教过的知识
- 你可以尝试基于已学内容举一反三，但要用不确定的语气，
  比如"我猜测..."或"根据你之前教我的，是不是..."
- 每次回答结束后，你应该向老师提出一个和学习相关的问题

你目前掌握的知识如下：

【已学规则】
1. 主语从句可以用what引导
2. 宾语从句中that有时可以省略
...（从 JSONB 中平铺生成）

【已学例句】
1. "What he said is true." —— what引导的从句作主语
2. "I think (that) he is honest." —— that引导的宾语从句
...（从 JSONB 中平铺生成）

如果上面的知识列表为空，说明你还什么都没学到，
请告诉老师你是一张白纸，期待他/她的教导。
```

### 7.2 知识提取 Prompt

用于从学生消息中提取知识点和例句，并自动判断所属分类。

```text
你是一个知识提取助手。请分析以下学生消息，提取其中关于英语名词从句的
知识点和例句。

学生消息："{student_message}"

请以 JSON 格式返回提取结果：
{
  "has_knowledge": true/false,
  "items": [
    {
      "type": "knowledge" 或 "example",
      "topic": "subject_clause / object_clause / predicative_clause
                / appositive_clause / general / other",
      "content": "知识点内容（type 为 knowledge 时）",
      "sentence": "例句（type 为 example 时）",
      "explanation": "例句解释（type 为 example 时）"
    }
  ]
}

分类说明：
- subject_clause：与主语从句相关
- object_clause：与宾语从句相关
- predicative_clause：与表语从句相关
- appositive_clause：与同位语从句相关
- general：跨类别的通用语法知识
- other：学习偏好、交流风格等非知识性内容

如果消息中没有可提取的知识（如寒暄、闲聊），返回 has_knowledge: false。
```

### 7.3 知识删除/修改 Prompt

当学生在对话中指出 AI 的某条记忆有误时触发。

```text
你是一个知识管理助手。学生指出 AI 的某条记忆需要修改或删除。

学生消息："{student_message}"

AI 当前的知识列表如下（含 ID）：
{knowledge_list_with_ids}

请判断学生想要修改或删除哪条知识，返回 JSON：
{
  "action": "delete" 或 "update",
  "target_id": "k_001",
  "new_content": "修改后的内容（仅 action 为 update 时）"
}

如果无法确定学生的意图，返回 {"action": "unclear"}。
```

### 7.4 Test 作答 Prompt

AI 参加选择题测试时使用。

```text
你是一个正在参加考试的AI学生。你只能基于你所学的知识来回答问题。

你目前掌握的知识如下：
{flat_knowledge_list}

请回答以下选择题，只选择一个答案。

题目：{question_text}
A. {option_a}
B. {option_b}
C. {option_c}
D. {option_d}

请以 JSON 格式返回：
{
  "answer": "A/B/C/D",
  "reasoning": "你选择这个答案的理由"
}

如果你没有学过相关知识，请根据有限的理解做出最佳猜测，
并在 reasoning 中说明你并不确定。
```

## 八、知识提取与更新流程

### 8.1 触发时机

每当学生在 Teach 对话中发送一条消息，系统自动触发一次知识提取。由于 DeepSeek API 成本较低，此频率不会产生显著费用。

### 8.2 处理流程

**Step 1**：学生发送消息，前端调用后端 `POST /api/chat/send` 接口。

**Step 2**：后端将学生消息发送给 DeepSeek API，使用"知识提取 Prompt"（7.2 节），获得结构化的提取结果。

**Step 3**：如果提取到新知识（`has_knowledge: true`），后端更新 ai_knowledge 表中对应记录的 JSONB 字段，将新知识条目追加到对应 topic 下，同时 version 递增。

**Step 4**：后端从数据库读取更新后的知识库，构造"Teach 对话 System Prompt"（7.1 节），连同本轮对话的历史消息（或摘要 + 最近若干轮消息），一起发送给 DeepSeek API 生成 AI 回复。

**Step 5**：AI 回复返回前端展示，同时后端将学生消息和 AI 回复都存入 chat_messages 表。

### 8.3 上下文管理

为避免对话过长导致 token 超限，当单次会话超过一定轮数（建议阈值 20 轮）时，后端自动调用 LLM 对历史消息进行摘要压缩，存入 session_summaries 表。后续对话的上下文构造方式变为：System Prompt + 摘要 + 最近 N 轮消息（建议 N=6）。

## 九、API 接口设计

### 9.1 认证模块

`POST /api/auth/register` —— 用户注册。请求体包含 username、password、display_name、role，以及可选的 invite_code。如果提供了 invite_code，注册完成后自动加入对应班级。

`POST /api/auth/login` —— 用户登录，验证成功后返回 JWT Token。

`GET /api/auth/me` —— 获取当前登录用户的基本信息（需携带 Token）。

### 9.2 班级管理（教师端）

`POST /api/classes` —— 创建班级，系统自动生成 6 位邀请码。

`GET /api/classes` —— 获取当前教师创建的所有班级列表。

`GET /api/classes/{id}` —— 获取班级详情，包括学生列表和邀请码。

`GET /api/classes/{id}/invite-qrcode` —— 生成并返回班级邀请二维码图片。二维码内容为注册页 URL，形如 `https://domain.com/register?code=ABC123`。

### 9.3 班级加入（学生端）

`POST /api/classes/join` —— 学生通过邀请码加入班级。后端同时为该学生创建一份空的 AI 知识库（基于班级的 knowledge_template）。

### 9.4 对话模块

`POST /api/chat/sessions` —— 创建新对话会话，需指定 class_id 和 session_type（teach/test）。

`GET /api/chat/sessions` —— 获取当前学生在某班级下的会话列表。

`POST /api/chat/send` —— 发送消息。请求体包含 session_id 和 content。后端执行知识提取 → 知识更新 → AI 回复生成的完整流程，返回 AI 的回复内容。

`GET /api/chat/sessions/{id}/messages` —— 获取某会话的全部消息记录。

### 9.5 知识库模块

`GET /api/knowledge/{class_id}` —— 获取当前学生在该班级的完整知识库 JSON（含分类结构，用于学生查看页面）。

`GET /api/knowledge/{class_id}/flat` —— 获取平铺格式的知识列表，不含分类标签（用于前端以卡片形式展示）。

`POST /api/knowledge/{class_id}/correct` —— 提交知识修正请求。后端调用知识删除/修改 Prompt 处理，返回操作结果。

### 9.6 测试模块（教师端）

`POST /api/tests/papers` —— 创建试卷，需指定 class_id 和 title。

`POST /api/tests/papers/{id}/questions` —— 为试卷添加选择题。

`PUT /api/tests/questions/{id}` —— 修改某道题目。

`DELETE /api/tests/questions/{id}` —— 删除某道题目。

`POST /api/tests/papers/{id}/execute` —— 对全班学生的 AI 执行测试。后端遍历班级中每位学生的 AI 知识库，逐题生成回答，计算得分，结果存入 test_results。

`GET /api/tests/papers/{id}/results` —— 获取该试卷的全部测试结果，按学生列出得分和每题详情。

### 9.7 数据分析（教师端）

`GET /api/analytics/class/{id}/overview` —— 班级整体知识覆盖率概览，统计各 topic 下有多少学生教授了知识。

`GET /api/analytics/class/{id}/students` —— 各学生的知识掌握详情，包含知识条目数量、测试得分等。

`GET /api/analytics/student/{id}/progress` —— 单个学生的学习进度时间线，展示知识积累和测试成绩变化。

## 十、前端页面设计

### 10.1 公共页面

**登录页**提供用户名密码输入框，登录后根据 role 自动跳转到对应的首页。**注册页**支持两种入口：手动注册（填写信息后手动输入邀请码加入班级）和扫码注册（通过二维码链接进入，邀请码自动填充）。

### 10.2 学生端页面

**学生首页**展示已加入的班级列表，每个班级卡片显示班级名称和 AI 的知识条目数量。页面顶部提供"输入邀请码加入新班级"的入口。

**AI 知识查看页**以更完整的 UI 展示 AI 的全部知识。知识以卡片网格或列表形式呈现，每张卡片显示知识内容、对应例句和学习时间。学生可点击某张卡片进入修正流程。

**试卷管理页**支持创建试卷、逐题添加四选一选择题、编辑和删除题目。创建完成后可点击"执行测试"按钮对全班 AI 进行测试。

**测试结果页**以表格形式展示每位学生的 AI 得分，支持展开查看每道题的 AI 回答和推理过程。可按题目维度横向对比各 AI 的答题情况。

## 十一、关键业务流程

### 11.1 学生注册与加入班级

教师创建班级后，系统自动生成 6 位邀请码（如 `ABC123`）和对应的二维码。二维码内容为指向注册页的链接（如 `https://domain.com/register?code=ABC123`）。教师将二维码通过课堂展示、微信群等方式分享给学生。学生扫码后进入注册页面，邀请码自动填充，学生只需设置用户名和密码即可完成注册并自动加入班级。系统在学生加入班级的同时，从班级的 knowledge_template 复制一份空的 JSON 结构，作为该学生的 AI 知识库初始值。

### 11.2 教学对话完整流程

学生进入 Teach 页面，选择开始新对话或继续已有对话。学生输入一条消息后，前端调用 `POST /api/chat/send`。后端首先调用 DeepSeek API 执行知识提取（使用 7.2 节的 Prompt），解析返回的 JSON 判断是否有新知识。如果有，将新知识条目追加到 ai_knowledge 表的 JSONB 中对应的 topic 下。随后，后端从数据库读取更新后的知识库，将所有非空知识条目平铺注入 System Prompt（使用 7.1 节的模板），连同对话历史一起发送给 DeepSeek API 生成 AI 回复。AI 回复返回后，前端展示给学生，后端同时将学生消息和 AI 回复存入 chat_messages 表。

### 11.3 教师测试完整流程

教师在试卷管理页创建试卷并添加选择题。点击"执行测试"后，后端遍历班级中每位学生对应的 AI 知识库。对每个学生，系统构造包含该 AI 知识的 Test Prompt（使用 7.4 节的模板），逐题发送给 DeepSeek API，收集 AI 的答案和推理过程。所有题目作答完成后，系统计算得分，将完整结果（含每题详情）以 JSONB 形式存入 test_results 表。教师可在测试结果页查看全班的成绩报告。

### 11.4 知识修正流程

学生在 AI 知识查看页或对话侧边栏发现某条知识有误，点击"标记为有误"。系统跳转到 Teach 对话页，并自动在输入框生成引导提示（如"我发现你关于 'xxx' 的理解似乎有误……"）。学生用自然语言描述问题所在，后端调用知识删除/修改 Prompt（7.3 节）识别目标条目和操作类型。后端执行相应的 JSONB 更新操作（删除或修改条目）。AI 在下一轮回复中确认"好的，我已经更正/忘掉了这条知识"。

## 十二、安全设计

### 12.1 密码存储

使用 bcrypt 对用户密码进行加盐哈希处理，数据库中仅存储哈希值，不保存明文密码。bcrypt 的 cost factor 建议设为 12。

### 12.2 认证机制

采用 JWT（JSON Web Token）进行无状态认证。Token 有效期设为 24 小时，Payload 中包含 user_id 和 role 字段。所有需要认证的 API 请求须在 HTTP Header 中携带 `Authorization: Bearer <token>`。Token 过期后前端自动跳转到登录页。

### 12.3 权限控制

后端在路由层实现角色权限校验。教师端接口（班级管理、试卷管理、数据分析）仅允许 `role=teacher` 的用户访问。学生端接口（对话、知识查看）仅允许 `role=student` 的用户访问，且后端校验该学生确实属于对应的班级，防止越权访问他人的知识库或对话记录。
